# Проверочный лист безопасности для бизнес-систем PropDevelopment

## Обоснование выбранных разделов

Для чеклиста выбраны разделы, которые напрямую связаны с текущими рисками PropDevelopment и системами на контейнерной диаграмме.

1. **Управление доступом и идентификацией**. В ландшафте есть несколько контуров аутентификации: Keycloak для клиентов и собственников, Active Directory для менеджеров, бизнес-аналитиков и финансовых систем. При этом в компании нарушена передача информации между командами, а специалист по ИБ не получает все изменения. Поэтому необходимо проверить централизованное управление доступами, MFA, пересмотр привилегий и разделение ролей.

2. **Безопасность данных и персональных данных**. Основная проблема компании - нарушенный контроль данных: несколько точек регистрации клиента, несистемный контроль внутренних потоков и отсутствие ограничений со стороны систем обработки данных. В системах обрабатываются ПДн клиентов и собственников, данные сделок, платежей, ЖКХ, объектов недвижимости и сырые данные DWH.

3. **Безопасность API и интеграций**. Наиболее явно описанная проблемная зона - API партнеров управляющих компаний: контракты содержат ПДн и могут позволять доступ к чужим клиентам и жилым комплексам. Также есть интеграции с платежной системой и государственными регистрационными органами.

4. **Защита инфраструктуры, сетей и контуров**. Большинство продуктов развернуто в приватном облаке и собственном ЦОД, часть - в гибридной инфраструктуре. Вход внешних пользователей идет через firewall, но для критичных подсистем важно проверить сегментацию, изоляцию баз данных, закрытые контуры для чувствительных данных и контроль облачных ресурсов.

5. **Резервное копирование и восстановление**. Для всех БД заявлено резервное копирование на необходимую глубину, но в описании нет подтверждения тестов восстановления, шифрования бэкапов и ограничения доступа к ним.

6. **Аудит, мониторинг и реагирование**. Компания готовится к комплексному аудиту после рыночного инцидента у конкурента, но ИБ-функция централизована в одном специалисте и получает не все изменения. Нужны проверки журналирования, SIEM/мониторинга, процесса реагирования и регистрации изменений.

7. **Data Governance для BI/ML/AI и DWH**. Компания хочет развивать BI-, ML- и AI-сервисы, но это заблокировано проблемами управления потоками данных. На диаграмме есть CDC-передача сырых данных в DWH/Greenplum и S3-хранилище, поэтому нужно отдельно проверить качество, происхождение, маскирование и доступ к данным.

## Приоритеты устранения выявленных пробелов

| Приоритет | Что исправить в первую очередь | Почему это важно |
| --- | --- | --- |
| P1 | Ввести единый реестр интеграций и обязательный security review API-контрактов партнеров | Это напрямую закрывает риск доступа партнеров к чужим клиентам, ЖК и избыточным ПДн. |
| P1 | Настроить объектную авторизацию и проверку владения данными в CRM, tenant-core-app и партнерских API | Это критично для устранения жалоб клиентов на отображение чужих данных и предотвращения утечек. |
| P1 | Ограничить доступ к ПДн в DWH/BI, прекратить неконтролируемую передачу сырых данных без маскирования | Без этого невозможно безопасно развивать BI/ML/AI и выполнять требования законодательства. |
| P2 | Ввести централизованный процесс согласования изменений с ИБ для потоков данных, прав доступа и внешних интеграций | Сейчас специалист ИБ не получает все изменения, из-за чего контроль безопасности фактически разрывается. |
| P2 | Проверить TLS/mTLS, шифрование данных на хранении, управление ключами и защиту резервных копий | В описании эти меры не подтверждены, хотя системы обрабатывают ПДн, платежные и сделочные данные. |
| P2 | Настроить централизованное журналирование и оповещения по критичным операциям с данными | Без журналов невозможно расследовать смешение профилей, несанкционированный доступ и действия партнеров. |
| P3 | Формализовать Data Governance: каталог данных, владельцы, lineage, качество данных, правила ML/AI | Это снижает риск некачественных и обесцененных данных и разблокирует развитие аналитических продуктов. |
