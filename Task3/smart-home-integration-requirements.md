# Требования к внешней интеграции сервисов Умный дом

## Цель и область изменений

PropDevelopment добавляет в мобильное приложение для собственников две новые функции:

- интеллектуальный домофон: видеозвонок, удаленное открытие двери, автоматическое открытие по распознаванию лица;
- интеллектуальный шлагбаум: удаленное управление и автоматическое открытие по распознаванию номера автомобиля.

Партнер предоставляет внешнюю платформу, которая управляет устройствами, биометрической идентификацией и распознаванием автомобильных номеров. Внутри PropDevelopment добавляется отдельный контейнер `smart-home-integration-app`, который изолирует партнерский API от основных бизнес-систем и выполняет авторизацию, минимизацию данных, аудит и контроль контрактов.

## Новые и измененные контейнеры

| Контейнер / система | Тип | Назначение | Ключевые данные |
| --- | --- | --- | --- |
| Витрина сервисов для собственников | Существующий mobile app | UI для функций домофона, шлагбаума, управления согласиями и разрешениями | JWT пользователя, статусы устройств, команды открытия |
| tenant-core-app | Существующий backend | Принимает команды из мобильного приложения, проверяет пользовательскую сессию, вызывает интеграционный слой | Идентификатор собственника, ЖК, дома, помещения, команды |
| CRM собственников | Существующий backend | Подтверждает связь пользователя с объектом недвижимости и управляющей компанией | Профиль собственника, помещение, дом, УК |
| smart-home-integration-app | Новый backend | Антикоррупционный слой для партнера: объектная авторизация, маппинг устройств, минимизация данных, rate limit, аудит | Внутренние идентификаторы, partner device id, разрешения, статусы, correlation id |
| smart-home-db | Новая БД | Хранит согласия, привязки устройств, ACL, журнал команд и событий | Согласия, разрешения, device mapping, audit trail |
| Smart Home Partner API | Внешняя система | Управляет домофонами, шлагбаумами, биометрией, распознаванием номеров и событиями устройств | Биометрические шаблоны, видео, номера авто, события доступа |
| SIEM / audit log | Существующий или целевой компонент | Централизованный аудит событий безопасности и пользовательских действий | Логи команд, отказов авторизации, изменений прав, ошибок API |

## Требования к безопасности

| № | Требование | Обоснование |
| --- | --- | --- |
| 1 | Внешняя платформа партнера не должна вызываться напрямую из мобильного приложения. Все вызовы проходят через `tenant-core-app` и `smart-home-integration-app`. | Исключает раскрытие партнерских ключей, снижает риск обхода бизнес-правил и сохраняет контроль PropDevelopment над данными. |
| 2 | Перед каждым действием с устройством должна выполняться объектная авторизация: пользователь, помещение, дом, подъезд, ЖК, управляющая компания, роль и срок действия разрешения. | В текущем ландшафте уже есть риск доступа партнеров и пользователей к чужим объектам; это основной контроль для его закрытия. |
| 3 | Для открытия двери или шлагбаума из приложения требуется step-up-аутентификация для рискованных сценариев: новое устройство, новый получатель доступа, удаленное открытие вне обычного паттерна, изменение биометрии или номера авто. | Команды физического доступа имеют высокий риск и требуют более сильного подтверждения личности. |
| 4 | Биометрия, видеопотоки и номера автомобилей не должны передаваться в DWH/BI и не должны использоваться для ML/AI без отдельного правового основания, согласия и обезличивания. | Эти данные относятся к чувствительным и могут привести к серьезным нарушениям законодательства и приватности. |
| 5 | PropDevelopment хранит только минимально необходимый набор данных: внутренний пользовательский ID, ID объекта, ID устройства партнера, статус согласия, разрешения и аудит. | Минимизация снижает последствия утечки и упрощает выполнение требований по ПДн. |
| 6 | Согласия на обработку биометрии и номеров авто должны быть явными, версионируемыми, отзывными и связанными с конкретной услугой и объектом. | Для биометрии и идентификаторов транспорта нужен контролируемый правовой контур обработки. |
| 7 | Все изменения разрешений доступа должны журналироваться с неизменяемым audit trail: кто, кому, к какому объекту, на какой срок, по какому основанию выдал или отозвал доступ. | Нужна расследуемость инцидентов физического доступа и ошибок авторизации. |
| 8 | Должны быть настроены лимиты на частоту команд открытия, регистрацию биометрии, добавление номеров авто и ошибки авторизации. | Снижает риск brute force, злоупотреблений и автоматизированных атак. |
| 9 | Контракты API партнера проходят security review до подключения и при каждом изменении версии. | В текущем описании именно партнерские API являются проблемной зоной. |
| 10 | В ответах API партнера запрещена передача лишних ПДн. Используется allowlist полей и схемная валидация входящих/исходящих сообщений. | Закрывает риск избыточного раскрытия данных между PropDevelopment и партнером. |
| 11 | Для всех команд открытия используется correlation id, который прокидывается от мобильного приложения до партнера и обратно в аудит. | Позволяет связать пользовательское действие, серверную авторизацию и событие устройства. |
| 12 | Секреты интеграции, клиентские ключи и сертификаты хранятся в защищенном secret storage, ротируются по регламенту и не попадают в код, логи и DWH. | Снижает риск компрометации партнерского API и массового управления устройствами. |
| 13 | Для интеграции требуется отдельная тестовая среда с синтетическими или обезличенными данными. | Команды могут использовать облачные ресурсы для тестов; нельзя переносить туда реальные ПДн и биометрию без контроля. |
| 14 | При недоступности партнера система должна переходить в безопасный режим: не выдавать новые разрешения и явно показывать статус ошибки, не выполняя повторные команды бесконечно. | Устройства управляют физическим доступом, поэтому ошибки интеграции не должны приводить к неконтролируемому открытию. |
| 15 | Партнер обязан передавать события доступа и административных действий обратно в PropDevelopment для аудита. | Без обратных событий невозможно расследовать открытия по биометрии, номеру авто или действия администратора партнера. |

## Аутентификация и авторизация

| Участок | Протокол / механизм | Требование |
| --- | --- | --- |
| Мобильное приложение -> auth-service-1 | OpenID Connect Authorization Code + PKCE | Пользователь получает JWT/refresh token через Keycloak. Для критичных действий применяется MFA или step-up. |
| Мобильное приложение -> tenant-core-app | HTTPS + Bearer JWT | `tenant-core-app` проверяет подпись, issuer, audience, срок действия токена и привязку пользователя к собственнику. |
| tenant-core-app -> smart-home-integration-app | Internal HTTPS + service JWT или mTLS | Только доверенный backend может вызывать интеграционный слой; пользовательский контекст передается как claims и correlation id. |
| smart-home-integration-app -> CRM собственников | Internal HTTPS + service account | Проверяется право пользователя на конкретный объект недвижимости, дом, подъезд и управляющую компанию. |
| smart-home-integration-app -> Smart Home Partner API | mTLS + OAuth2 Client Credentials | Используются отдельные client credentials на среду и партнера; scopes ограничены конкретными операциями: `device.read`, `access.command`, `access.grant`, `event.read`. |
| Partner API -> smart-home-integration-app webhook | mTLS + подпись события HMAC/JWS | Все события партнера проверяются по сертификату, подписи, timestamp и nonce для защиты от replay. |
| Администрирование доступов УК | RBAC/ABAC в PropDevelopment | Сотрудник УК или менеджер видит только свои ЖК, дома и помещения; действия требуют аудита и, для массовых изменений, дополнительного согласования. |

## Организация взаимодействия

### Открытие домофона или шлагбаума из приложения

1. Собственник входит в мобильное приложение через Keycloak.
2. Мобильное приложение отправляет команду в `tenant-core-app` с JWT и correlation id.
3. `tenant-core-app` проверяет сессию и передает команду в `smart-home-integration-app`.
4. `smart-home-integration-app` запрашивает CRM собственников и проверяет, что пользователь имеет право управлять конкретным домом, подъездом, помещением или шлагбаумом.
5. `smart-home-integration-app` проверяет локальные разрешения, согласия, лимиты и состояние устройства в `smart-home-db`.
6. Если проверка успешна, `smart-home-integration-app` вызывает Partner API по mTLS и OAuth2 Client Credentials.
7. Партнер выполняет команду на устройстве и возвращает результат.
8. `smart-home-integration-app` записывает команду, ответ партнера и итоговый статус в audit log и возвращает результат пользователю через `tenant-core-app`.

### Автоматическое открытие по лицу или номеру автомобиля

1. Собственник в приложении выдает согласие на обработку биометрии или номера автомобиля.
2. `smart-home-integration-app` регистрирует согласие и передает партнеру только данные, необходимые для конкретной услуги.
3. Партнер обрабатывает биометрию или номер автомобиля в своей платформе и управляет устройством.
4. Партнер отправляет в PropDevelopment событие доступа: устройство, время, результат, тип идентификации, обезличенный идентификатор субъекта, correlation/event id.
5. `smart-home-integration-app` проверяет подпись события, сохраняет его в аудит и при необходимости обновляет статус в мобильном приложении.

### Выдача временного доступа гостю

1. Собственник выбирает объект, устройство, срок и тип доступа.
2. Система проверяет, что собственник может выдавать гостевой доступ для этого объекта.
3. Для создания или продления доступа применяется step-up-аутентификация.
4. `smart-home-integration-app` создает ограниченное разрешение в `smart-home-db` и передает партнеру только технический идентификатор доступа и срок.
5. По истечении срока доступ автоматически отзывается; событие отзыва журналируется.

## Контроль данных

| Категория данных | Где обрабатывается | Правило |
| --- | --- | --- |
| Профиль собственника, объект недвижимости, связь с УК | CRM собственников, tenant-core-app | Не передавать партнеру Ф. И. О. и контактные данные без необходимости; для команд использовать технические идентификаторы. |
| Биометрические шаблоны лица | Платформа партнера | Не хранить в DWH/BI; хранить и обрабатывать только при наличии явного согласия и договора с партнером. |
| Видеопоток домофона | Платформа партнера, мобильное приложение по защищенному каналу | Не логировать содержимое видеопотока; фиксировать только метаданные события. |
| Номера автомобилей | Платформа партнера, `smart-home-db` при необходимости | Хранить в PropDevelopment только если это требуется бизнес-процессом; применять маскирование в логах. |
| Команды открытия и события доступа | `smart-home-db`, SIEM/audit log | Хранить как события безопасности с неизменяемым аудитом и сроком хранения по регламенту. |
| Секреты интеграции | Secret storage / KMS | Не хранить в БД приложения, репозитории, логах и DWH. |

## Требования к эксплуатации и контролю

| Область | Требование |
| --- | --- |
| Мониторинг | Настроить метрики доступности Partner API, ошибок авторизации, отказов устройств, задержек команд, превышения лимитов и webhook replay. |
| Оповещения | Создать алерты на массовые открытия, частые отказы доступа, изменение большого числа разрешений, вызовы API вне allowlist, ошибки проверки подписи webhook. |
| Журналирование | Логи должны содержать user id, object id, device id, action, result, source system, partner event id, correlation id; ПДн в логах маскируются. |
| Версионирование API | Поддерживать версионированные контракты, backward compatibility и процедуру отключения устаревших версий. |
| Тестирование безопасности | Перед вводом выполнить threat modeling, SAST/DAST, проверку контрактов API, тесты объектной авторизации и негативные тесты на доступ к чужим ЖК. |
| Правовые требования | Заключить договор обработки данных с партнером, определить роли оператора/обработчика, хранение данных, сроки удаления, порядок отзыва согласия и уведомления об инцидентах. |
| Инциденты | Зафиксировать playbook для компрометации партнерского ключа, утечки биометрии, ошибочного открытия устройства, массовой выдачи доступов и недоступности партнера. |
| Резервное восстановление | Бэкапировать `smart-home-db`, регулярно проверять восстановление, но не восстанавливать отозванные согласия и просроченные разрешения как активные без дополнительной сверки. |
